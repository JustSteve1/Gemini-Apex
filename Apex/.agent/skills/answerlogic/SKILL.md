# answerlogic Skill

## Instructions

When writing SQL based on the Conceptual Graph generated by the `question2query` skill, you MUST abide by the following business logic rules to map the requested data to the correct Google Cloud BigQuery tables. You MUST verify all table and column names exist in your live schema before writing the query.

### Rule 1: Default Context
If no specific `time`, `date`, `year`, or `session_type` parameter is provided in the extracted entities from the user's question, you **MUST ASSUME** the user is referring to the most recent session.

Because `telemetry` is a massive unpartitioned table (400MB+), you MUST NOT use a dynamic `ORDER BY date DESC` or `MAX(date)` query to find the latest session, as scanning the whole table will exceed the maximum 1GB query byte limit and crash.

Instead, when no year is provided, you MUST explicitly hardcode `WHERE season = 2026 AND event_name = 'Australian Grand Prix'` into your query to reduce the data scanned. This is the most recent active session in the database.

### Rule 2: All Lap and Telemetry Data
For **ALL** lap time requests (e.g., "fastest lap", "lap times") AND telemetry requests (e.g., "speed", "throttle trace", "brake", "gear", "rpm", "drs"), you MUST exclusively query the `f1-data-library_DS.telemetry` table.

Do NOT use `public_laps`, `laps`, or `public_telemetry_intervals_enriched`. The `telemetry` table contains the central source of truth for both `lap_time_sec` and `speed`/`throttle`/etc.

CRITICAL LAP TIME LOGIC: In the `telemetry` table, `lap_time_sec` is a continuous timer for every telemetry sample, meaning it starts at 0.000 for every lap. You CANNOT simply `ORDER BY lap_time_sec ASC` to find the fastest lap, as that will return 0.000. 
To find the fastest or completed lap times, you MUST always group by `driver_number` and `lap_number`, calculate the `MAX(lap_time_sec)` to find the lap's full duration, and order by that duration. Example:
```sql
WITH LapDurations AS (
    SELECT driver_number, lap_number, MAX(lap_time_sec) AS lap_time_sec
    FROM f1_data_library_DS.telemetry
    WHERE season = 2026 AND event_name = 'Australian Grand Prix' AND session_type = 'Q'
    GROUP BY driver_number, lap_number
)
SELECT driver_number, lap_time_sec FROM LapDurations ORDER BY lap_time_sec ASC LIMIT 1
```

CRITICAL DATA LIMIT REQUIREMENT: Because the telemetry table is massive, you MUST ONLY SELECT the exact columns you need to answer the prompt (e.g., `SELECT driver_number, lap_time_sec`). NEVER use `SELECT *` or retrieve unneeded positional columns like `x, y, z`, as this will immediately exceed the data scan limit and crash the query.

### Rule 3: Events, Races, and Sessions
The `telemetry` table natively includes the `event_name` (e.g. "Australian Grand Prix"), `session_type` (e.g., 'Q', 'R', 'FP1'), and `season` (e.g., 2026). You do NOT need to join the `public_races` or `public_sessions` tables. Just filter the `telemetry` table directly.

CRITICAL: The `session_type` strings are abbreviated. You MUST USE these exact strings and NEVER use the full word:
- Qualifying = `'Q'`
- Race = `'R'`
- Practice = `'FP1'`, `'FP2'`, `'FP3'`

### Rule 4: Driver Information
The `telemetry` table only has the string `driver_number` (e.g., '1', '44'). 
To get a driver's name (e.g., full name, or code like 'VER'), you MUST join the `driver_mapping_2026` table like this:

```sql
JOIN f1_data_library_DS.driver_mapping_2026 AS dm
  ON TRIM(CAST(t.driver_number AS STRING)) = TRIM(CAST(dm.driver_number AS STRING))
```

Use `dm.name_acronym` as the driver code (e.g. 'VER') and `dm.full_name` as the full name.
Do NOT use `public_driver_number_mapping` or `public_drivers`.

### Rule 5: Formatting Lap Times
In the `telemetry` table, laps are stored as `lap_time_sec` (FLOAT64). BigQuery's `FORMAT()` requires an `INT64`.

To format `lap_time_sec` into a readable MM:SS.mmm format (`1:23.456`), you MUST copy and use this exact SQL snippet without modification:
```sql
FORMAT("%02d:%02d.%03d",
    CAST(FLOOR(lap_time_sec / 60) AS INT64),
    CAST(FLOOR(MOD(CAST(lap_time_sec AS NUMERIC), 60)) AS INT64),
    CAST(MOD(CAST(lap_time_sec * 1000 AS INT64), 1000) AS INT64)
) AS formatted_lap_time
```
